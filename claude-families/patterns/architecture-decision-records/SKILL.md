# Architecture Decision Records

Record important decisions while the reasoning is still fresh.

## Good ADR Candidates

- framework or database choice
- API style or boundary decisions
- deployment or infrastructure choices
- security strategy
- testing or release-process decisions

## ADR Shape

- title
- date and status
- context
- decision
- alternatives considered
- consequences
- risks

## Rules

- record the why, not just the what
- include rejected alternatives
- keep it short enough to reread easily
- do not create ADRs for trivial choices

Use ADRs to reduce future “why did we do this?” archaeology.
